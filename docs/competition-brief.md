# Competition Brief

## 题目总结

项目选题：AI 小说转剧本工具

核心要求：
- 输入不少于 3 个章节的小说文本
- 自动输出结构化剧本初稿
- 输出格式必须是 YAML
- 结果应可编辑、可进一步打磨，而不是一次性黑盒成品
- 需要额外提供一份“剧本 YAML Schema 定义文档”，并解释 Schema 设计原因

一句话目标：
- 做一个以“后端生成管线 + 结构化输出 + 可编辑初稿”为核心价值的改编工具，而不是只做普通对话式摘要器

## 评审重点

权重拆解：
- 作品完整度与创新性：40%
- 开发过程与质量：40%
- 演示与表达：20%

从评审标准倒推的项目策略：
- 代码不只要能跑，还要能被快速理解和复现
- 需要明确展示工程能力、仓库协作能力、PR 粒度和提交节奏
- 需要让评委或 AI agent 在短时间内看懂“系统做了什么、为什么这样设计、当前完成到哪里”
- demo 不只是展示页面，还要能讲清后端生成链路、结构化产物、可扩展性

## 精简后的硬性要求

作品提交物：
- 公开可访问的代码仓库
- demo 视频
- README 文档

有效性要求：
- 整个 72h 周期内保持持续提交，不能只在最后一天一次性导入代码
- commit 时间戳必须落在报名批次的起止时间内
- PR 描述不能为空，且需与实际代码变更相符
- 使用第三方依赖或框架时，README 中必须说明
- 若复用过往代码，需要在 PR 描述中注明来源

PR 规范：
- 基于 PR 交付新功能
- 每个 PR 只做一件事
- 大功能必须拆分成多个小 PR
- PR 标题和描述必须写清楚：做了什么、为什么做、如何实现、如何测试
- 主分支在任何时刻都应保持可运行或可验证状态

Demo 视频：
- 需要完整讲解主要功能与效果
- 链接必须放在仓库 README 的显眼位置

## 团队背景与项目导向

当前队伍分工：
- 后端：你，Go 为主，Python 仅在必要处少量使用
- 前端：队友负责

因此项目导向应当是：
- 后端主导：核心竞争力来自文本处理、结构化生成、YAML 校验、任务编排、输出稳定性
- 前端轻但完整：聚焦输入、任务进度、结果查看、结果编辑和导出
- 单仓库协作：前后端明确目录分离，减少冲突与沟通成本

## 针对 AI 评审与竞争环境的补充判断

考虑到评委可能使用 AI agent 快速筛仓：
- 文档必须先于代码清晰表达目标、边界、架构和进度
- 目录结构必须一眼可读
- YAML Schema 必须显式、稳定、能被程序校验
- PR 和 commit 需要能体现连续开发与工程思维，而不只是功能堆砌

考虑到与其他参赛者竞争：
- 只做“LLM 聊天改编”不够，需要突出结构化产物和工程化管线
- 后端需要有明确的多阶段 pipeline 设计，而不是把所有逻辑写进一个接口
- 需要保留未来可扩展点，例如章节切分、人物抽取、场景规划、YAML 验证、导出和重试机制

## 本仓当前落地原则

为了让后续 session 不偏题：
- 所有架构判断优先依据 `docs/` 文档
- 所有进度定位优先依据 `implementation-progress.md`
- 所有功能补全优先依据 `final-solution.md` 与 `frontend.md`
- 未被文档确认的大改动，不应直接实施
