# Frontend Visual Direction

## 用途

本文件锁定前端首版的视觉方向、信息优先级和交互基调。

目标不是把所有像素做死，而是让后续 human/agent session 在不反复猜测审美方向的前提下，稳定实现一个：
- 看起来像“创作工具”而不是通用后台
- 适合中文长文本录入与 YAML 查看
- 能体现“小说转剧本”的编辑感和工程感

## 视觉定位

首版视觉定位固定为：
- `Editorial Studio`
- `Writer's Desk`
- `Structured Drafting Tool`

不要做成：
- 通用 CRM / Admin Dashboard
- 过度霓虹或赛博风
- 纯聊天工具界面
- 过重拟物、过度影视海报化界面

一句话方向：
- 像一张正在被编排的“创作工作台”，而不是一个普通表单页

## 视觉关键词

必须体现：
- 清晰
- 稳定
- 有编辑感
- 有轻微戏剧气质
- 适合长文本停留

允许出现：
- 纸张感
- 页边批注感
- 时间线或分镜感
- 轻量“排演室”氛围

不允许出现：
- 紫色默认科技感
- 纯黑重暗黑模式倾向
- 玻璃拟态堆叠
- 大量渐变按钮和营销页风格

## 色彩方案

默认色板固定如下：

### Base

- `--bg-page: #f5f1e8`
- `--bg-panel: #fbf8f1`
- `--bg-muted: #ece5d8`
- `--border: #d6c9b3`
- `--text-primary: #2b241d`
- `--text-secondary: #5b5146`

### Accent

- `--accent-primary: #8b3f2f`
- `--accent-primary-hover: #713224`
- `--accent-secondary: #2f5d62`
- `--accent-highlight: #b88b3b`

### Status

- `--success: #3d6b4f`
- `--warning: #a06a1c`
- `--error: #9f3f3f`
- `--info: #46637a`

使用原则：
- 主背景走暖灰纸面色
- 主按钮和关键强调使用砖红色系
- 状态信息、辅助标签允许使用深青色
- 高亮或步骤标识可用少量黄铜色

## 字体策略

默认字体策略：
- 中文标题：`"Noto Serif SC", "Source Han Serif SC", serif`
- 中文正文与 UI：`"Noto Sans SC", "Source Han Sans SC", sans-serif`
- 等宽区（YAML / stage ids / request ids）：`"IBM Plex Mono", "JetBrains Mono", monospace`

约束：
- 不使用 `Inter` 作为主视觉字体
- 不使用默认系统 sans 作为唯一字体
- YAML 编辑区必须使用等宽字体

## 布局方案

桌面端首版默认使用双栏工作台：

### 左栏 `Input Workspace`

宽度倾向：
- `34%` 左右

内容顺序：
1. 作品标题与作者
2. 改编风格与补充要求
3. 章节列表
4. 生成按钮

视觉特征：
- 像“创作素材板”
- 允许卡片分节
- 章节块之间要有明确编号感

### 右栏 `Result Workspace`

宽度倾向：
- `56%` 左右

内容顺序：
1. 顶部生成进度条
2. 结果头部操作条
3. YAML 文本编辑区
4. 结构化摘要区

视觉特征：
- 是页面重心
- 有“剧本初稿”阅读感
- 不要做成纯代码编辑器页面
- 进度条应并入结果区，不再独立成一整列

### 移动端

移动端默认改为纵向堆叠：
1. 输入区
2. 结果区

若时间允许，可加吸顶切换：
- `Input`
- `Result`

但这不是首版 blocker。

## 组件视觉约束

### Panel

所有主区块统一使用：
- 浅暖底色
- 大圆角
- 色块层级
- 轻阴影或纸张边缘感

不要使用：
- 满屏 1px 边框套边框
- 生硬直角
- 夸张毛玻璃

### Button

按钮优先级：
- 主按钮：砖红底，浅色字
- 次按钮：浅色实体底，不靠明显边框区分
- 危险按钮：仅错误或清空操作使用

生成按钮必须视觉上最明确。

### Form

输入区要求：
- 标签始终显示，不依赖 placeholder 代替
- 长文本输入必须有足够高度
- 章节卡片必须清晰显示章节序号

### Timeline

阶段状态推荐为结果区顶部的紧凑双行进度条：
- 已完成：点位与连线填色
- 当前阶段：强调色 + 轻脉冲
- 未开始：低对比点位与胶囊
- 失败：错误色 + 错误说明

### YAML Editor

YAML 区域要求：
- 等宽字体
- 高度优先给足
- 行距不要太密
- 与结构化摘要区有明确分隔

不要求首版有复杂语法高亮，但要可读。

## 信息优先级

首屏优先让评委看到：
1. 这是一个“小说转剧本”的工具
2. 输入区支持多章节
3. 结果区顶部能看到任务阶段
4. 右侧有合法 YAML 结果

所以页面首屏必须有：
- 作品与章节输入
- 明确的“生成”动作
- 可见的阶段状态
- 可见的 YAML 或结构化结果

不要把关键信息折叠得太深。

## 结果区展示规则

结果区分两层：

### 顶层操作条

必须有：
- 当前 job id 或简化标识
- 导出按钮
- 重置为后端原始结果按钮

可选：
- 复制 YAML

### 内容区

默认上下结构：
1. YAML 编辑区
2. 结构化摘要区

结构化摘要区默认内容顺序：
1. 角色列表
2. 地点列表
3. scene 卡片列表

scene 卡片最少展示：
- scene title
- source chapter
- slugline
- objective
- open question

## 动效策略

只允许少量有意义的动效：
- 页面初次进入时卡片轻微上浮
- job 当前阶段有轻脉冲或进度强调
- 新结果载入时结果区淡入

禁止：
- 大面积连续动画
- 过多 hover 特效
- 跳动式 loading

## 文案风格

文案要求：
- 简短
- 工具型
- 不要过度营销化

建议用词：
- `生成剧本草稿`
- `任务处理中`
- `生成进度`
- `导出 YAML`
- `恢复后端原始结果`

不建议：
- `AI 魔法生成`
- `一键爆改神作`
- 夸张宣传文案

## 前端实现自由度边界

前端 teammate 或 agent 可以自由细调：
- 字号层级
- 间距
- 卡片排列
- 响应式折叠方式
- 摘要区具体排版

但不得擅自改变：
- 三大工作区结构
- 任务化 API 路径
- YAML 作为结果区核心
- 页面首屏信息优先级
- 当前暖纸面 editorial 风格方向

## 验收标准

当前视觉方向达标标准：
- 一眼看得出是创作工具，不是后台系统
- 输入、状态、结果三区结构清晰
- YAML 区域可读
- 中文长文本阅读不累
- 在桌面和移动端都能正常使用
