# Demo 录制指导

## 用途

本文件是当前版本 ScriptForge 的 demo 录制字稿与镜头顺序说明。

目标只有两个：
- 让评委快速看懂这不是普通聊天式生成页，而是“小说 -> 结构化剧本 YAML 初稿”的工作台
- 让演示内容和当前页面、当前后端主链路、当前 README 叙事完全一致

## 当前版本结论

当前版本已经可以直接录主视频，推荐使用：
- 样例：`《雾港回声》`
- 生成方式：`AI 增强`
- 路径：`切换为空白手工输入 -> 粘贴真实三章 -> 生成剧本初稿 -> 等待右侧自动载入结果`

当前页面已经足以支撑完整讲解，原因是：
- 左侧 `创作素材台` 明确体现了“3 章以上输入”和改编要求整理
- 右侧 `生成进度` 明确体现了 job pipeline，不是伪造 loading
- 右侧同时保留 `本次结果`、`YAML 剧本初稿`、`角色、地点与场景`，而且结果提醒区已经压成更紧凑的状态提示，不会喧宾夺主

## 当前页面标题对照

录制时请直接按页面上的实际标题讲，不要再使用旧文案：

- 页面主标题：`ScriptForge 剧本改编工作台`
- 左侧列标题：`创作素材台`
- 右侧列标题：`生成结果`
- 右侧顶部进度：`生成进度`
- 结果工具区：`本次结果`
- YAML 区：`YAML 剧本初稿`
- 摘要区：`角色、地点与场景`
- 左侧空白录入入口：`切换为空白手工输入`
- 主按钮：`生成剧本初稿`

不要在录制里说：
- `Input Workspace`
- `Result Workspace`
- “中间状态列”

这些都不是当前页面形态。

## 录制主叙事

这个项目不应该被讲成：
- “我们接了一个模型接口，可以把小说转成剧本”

应该被讲成：
- “我们做了一个面向小说改编的结构化剧本工作台。输入是 3 章以上小说文本，后端会走任务化 pipeline，最后产出可继续编辑的 YAML 剧本初稿。”

评委最应该记住的 4 件事：
1. 输入不是一句 prompt，而是 `3 章以上小说`
2. 输出不是一段普通文本，而是 `YAML 剧本初稿`
3. 生成不是同步黑盒返回，而是显式 job pipeline
4. 结果不是最终成片，而是可编辑、可恢复、可导出的结构化初稿

## 推荐视频时长

- `2.5` 到 `4` 分钟

## 推荐镜头顺序

1. `15-20s` 开场，对齐题目与差异点
2. `40-60s` 左侧 `创作素材台`
3. `30-45s` 点击 `生成剧本初稿` 后讲右侧 `生成进度`
4. `50-70s` 讲 `本次结果` 和 `YAML 剧本初稿`
5. `40-60s` 讲 `角色、地点与场景`
6. `20-30s` 演示恢复、复制、导出
7. `20-30s` 收尾，点明后端工程价值

## 可直接照读的开场

建议开场表述：

“这个项目做的是 AI 小说转剧本工具，但它不是普通聊天式生成。我们要求输入至少 3 个章节的小说文本，后端把整个改编过程拆成一个真实的任务化 pipeline，最后输出可继续编辑的 YAML 剧本初稿。作者拿到结果后，不只是看一段文案，而是可以继续编辑、恢复和导出。”

## 镜头逐段字稿

### 第一段：左侧 `创作素材台`

镜头停留位置：
- 左侧整列

必须讲的点：
- 这里要求 `至少 3 章`
- 可以直接用示例，也可以点 `切换为空白手工输入`
- 当前主视频用真实小说《雾港回声》，不是仓库内置短样例

建议讲法：

“左侧是创作素材台。这里不是单轮 prompt，而是按作品标题、改编风格、目标受众和多章节正文组织输入。当前演示我不用内置示例，而是切到空白手工输入，直接粘贴《雾港回声》的三章真实文本，证明主链路不依赖 preset。”

这一段不要花太久时间在：
- 视觉样式
- 表单小交互

### 第二段：点击 `生成剧本初稿`

镜头停留位置：
- 点击主按钮
- 立刻切到右侧 `生成结果`

必须讲的点：
- 点击后不是同步返回
- 前端先调用 `POST /api/v1/jobs`
- 之后轮询 `GET /api/v1/jobs/{jobID}`
- 生成完成后再拉 `GET /api/v1/jobs/{jobID}/result`

建议讲法：

“点击生成剧本初稿之后，前端不是等待一个同步大响应，而是先创建任务。也就是先调 `POST /api/v1/jobs`，拿到 job id 之后持续轮询 `GET /api/v1/jobs/{jobID}`，等任务完成后再读取结构化结果。这说明它不是一个同步黑盒接口，而是真实的 job pipeline。”

### 第三段：右侧 `生成进度`

镜头停留位置：
- 右侧顶部 `生成进度`

当前页面上的阶段名就是：
- `素材接收`
- `章节梳理`
- `角色地点`
- `场景规划`
- `剧本生成`
- `结构校验`
- `结果保存`

建议讲法：

“这里能直接看到后端不是一次性生成，而是拆成了素材接收、章节梳理、角色地点、场景规划、剧本生成、结构校验和结果保存这些阶段。页面上的进度不是装饰，而是对应真实 job 的处理状态。”

录制时要注意：
- 不要承诺“每个阶段都会细粒度实时跳动”
- 当前实现里阶段写回仍偏粗，讲成“真实轮询中的处理状态”就够了
- 重点是任务化和阶段化，不是动画精度

### 第四段：右侧 `本次结果`

镜头停留位置：
- `本次结果`

必须讲的点：
- 这里会显示本次 `job id`
- 会明确区分 `当前为生成初稿` 和 `当前为本地编辑稿`
- 这里的操作按钮证明结果不是静态截图

建议讲法：

“任务完成后，右侧先进入本次结果区。这里会显示当前 job 编号，也会区分现在展示的是生成初稿，还是我在页面里做过本地修改后的版本。下面这些按钮分别对应恢复生成初稿、下载后端原始 YAML、复制当前 YAML 和导出当前 YAML。”

### 第五段：右侧 `YAML 剧本初稿`

镜头停留位置：
- `YAML 剧本初稿`

讲解顺序必须是：
1. 先讲 YAML
2. 再讲下面的摘要

建议讲法：

“这里先看 YAML 剧本初稿，因为这才是项目的权威产物。它不是一段自然语言回复，而是结构化剧本草稿。作者后续可以直接在这里继续改，也可以复制和导出。整个项目强调的是 YAML-first，而不是把一段模型输出硬塞进页面。”

必须点出的后端亮点：
- 输出契约是 YAML
- YAML 会经过 schema 约束和结构校验
- 前端不是自己瞎解析文本，而是接收后端返回的结构化结果

### 第六段：右侧 `角色、地点与场景`

镜头停留位置：
- `角色、地点与场景`

建议讲法：

“YAML 下面这一块是结构化摘要区，方便快速回看整份初稿的规模和结构。这里会展示章节数、场景数、角色数、结构校验状态，以及角色、地点和场景卡片。它的作用是辅助浏览，不是替代 YAML 本身。”

这里要主动补一句：

“这个区块上方现在会先给一个很短的复核提示，只告诉你这还是可编辑初稿、结构是否通过、以及这次有没有额外提醒。它不会再塞一整块固定说明，但依然会把真实 warning 留出来，避免假装已经完成终稿。”

### 第七段：编辑、恢复、复制、导出

建议操作：
1. 在 YAML 中改一两行
2. 展示 `当前为本地编辑稿`
3. 点一次 `恢复生成初稿`
4. 点一次 `复制当前 YAML` 或 `导出 YAML`

建议讲法：

“这里的定位不是一次性成片，而是作者可继续打磨的结构化初稿。所以我可以直接改 YAML，系统会把状态切换到本地编辑稿；如果我想回到后端本次生成的版本，也可以一键恢复生成初稿。最后还能复制或导出，继续进入后续工作流。”

## 录制时必须主动讲出的后端亮点

前端不会自动替你说清这些点，必须自己讲出来：

1. 后端是任务化 API，不是同步生成接口
2. 后端有显式 pipeline，而不是单接口包办一切
3. 结果是 YAML-first，不是普通长文本
4. `AI 增强` 是主路径，`标准草稿（兜底）` 只是 fallback / smoke baseline
5. 结果会经过 normalize、validation 和 schema 约束
6. provider 走的是 `openai_compatible` 抽象，不绑死单一家
7. job 结果和 artifact 有持久化，不是一次性内存 demo

## 主视频推荐配置

推荐主视频：
- 样例：`《雾港回声》`
- 页面选择：`切换为空白手工输入`
- 生成方式：`AI 增强`

推荐理由：
- 最贴合比赛题意
- 最能证明真实三章节输入主链路已经跑通
- 最能避免被评委误解为“只对内置样例有效”

备选保底视频：
- 样例：`职场`
- 生成方式：`AI 增强`

用途：
- 当你只想更快录一次更稳的演示时使用

## 备选补充镜头：失败可恢复

如果你还想录一个补充镜头，可以单独补这一段，不建议塞进主视频主体：

前置条件：
- 后端使用 `LLM_PROVIDER=disabled`
- 页面仍选择 `AI 增强`

录法：
1. 提交一次生成
2. 展示任务最终仍能完成
3. 指向页面里的提醒文案，说明这次不是实际 LLM 成果，而是显式回退到了兜底路径
4. 再点一次 `重新生成当前内容`

这段的作用：
- 证明 provider 不可用时系统不会静默伪装成真实 AI 结果
- 证明前后端的回退与重新生成链路是打通的

注意：
- 录制时不要对评委念内部字段名，比如 `validation.warnings`
- 直接讲“页面会明确提醒这次不是实际 AI 生成结果”即可

## 不建议的录法

不要这样录：
- 一上来先讲 UI 漂亮
- 全程只讲交互，不讲后端任务模型
- 只展示最终结果，不展示处理中状态
- 先讲摘要卡片，后讲 YAML
- 把项目讲成“接了个模型接口自动生成”
- 把 `标准草稿（兜底）` 讲成主要能力

这些录法会直接削弱当前仓库最有价值的部分，也就是结构化输出和工程化后端。

## 录制前检查清单

录制前至少确认：
- 后端 `:8080` 可启动
- 前端 `:5173` 可启动
- `《雾港回声》` 三章文本已准备好
- `切换为空白手工输入 -> 粘贴三章 -> 生成剧本初稿` 这条路径可直接跑通
- 右侧能正常出现 `生成进度`、`本次结果`、`YAML 剧本初稿`、`角色、地点与场景`
- `恢复生成初稿`、`复制当前 YAML`、`导出 YAML` 可以正常演示
- 若要录失败补充镜头，再额外确认 disabled-provider fallback 提示链路可见

## 建议收尾

建议收尾表述：

“所以这个项目的重点不是做一个会写点文本的 AI 页面，而是把小说改编拆成一个可解释、可验证、可编辑、可导出的剧本生成流程。前端负责把这条链路呈现出来，真正的竞争力主要在后端的结构化 pipeline 和工程落地能力。”
