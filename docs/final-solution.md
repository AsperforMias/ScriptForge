# Final Solution

## 最终目标

在 72 小时内交付一个“AI 小说转剧本工具”的可演示 MVP：
- 支持输入不少于 3 个章节的中文小说文本
- 自动生成结构化剧本 YAML 初稿
- 支持作者查看、编辑、导出 YAML
- 明确展示后端生成管线、结构化设计能力和团队协作工程质量

## 产品边界

本项目要解决的问题：
- 让小说作者快速获得可继续打磨的剧本草稿
- 将“长文本改编”从一次性对话输出，升级为结构化、多阶段、可验证的生成流程

本项目不追求的内容：
- 不追求电影工业级终稿
- 不追求复杂账号体系、支付、多人实时协作
- 不追求一次性支持多种剧本标准
- 不在首版追求重量级基础设施，如分布式队列、复杂云原生部署

## MVP 范围

MVP 必做：
- 小说文本输入：支持粘贴或上传 3 章以上文本
- 基础元信息：作品名、章节标题、改编风格等
- 后端生成管线：章节解析、人物/场景抽取、剧情拆解、场景级剧本生成、YAML 组装、Schema 校验
- 任务化处理：创建任务、查询状态、获取结果、导出 YAML
- 结果展示：前端查看 YAML 与结构化摘要
- 结果编辑：允许对 YAML 文本进行人工修改
- 结果导出：下载 YAML 文件
- 样例数据与回归用例：至少一组样例输入和对应输出

MVP 可选增强：
- 生成过程日志或阶段状态
- 同步重试或局部重生成
- 场景卡片视图
- 人物表和场景表辅助编辑

明确不做：
- 富文本剧本编辑器
- 多租户用户系统
- 高级权限系统
- 云端长任务编排平台

## 架构形态

采用单仓前后端分离、后端管线主导的架构：

```text
frontend (teammate-owned UI)
    -> backend HTTP API (Go)
        -> middleware chain
        -> in-process job runner
        -> ingest / normalize
        -> outline extraction
        -> character & location extraction
        -> scene planning
        -> screenplay YAML generation
        -> schema validation
        -> artifact persistence
```

关键架构选择：
- 后端主语言使用 Go，突出服务化、模块化和可维护性
- Python 仅在必要的实验性文本处理环节使用，并保持边界清晰；默认不作为主链路依赖
- 服务形态采用单体 HTTP API + 进程内异步任务执行器，避免过度工程化，同时保留任务状态和阶段追踪
- 数据持久化使用 SQLite + 本地文件产物目录，保证 72h 内可交付、易演示、易复现
- 生成结果以 YAML 为核心产物，避免只保留不可追踪的自然语言输出
- Go 中间件栈显式暴露请求 ID、恢复、超时、结构化日志、体积限制和 CORS，体现服务端工程能力
- 管线按阶段拆分，便于展示工程能力、测试边界和后续扩展

## 模块职责

`backend/cmd/api`
- 服务入口
- 配置加载
- HTTP 路由和依赖装配

`backend/internal/ingest`
- 接收小说章节输入
- 章节清洗、标准化、长度校验
- 检查“至少 3 章”的题目硬约束

`backend/internal/pipeline`
- 串联整个生成流程
- 管理阶段状态、错误传播和结果汇总
- 驱动同步逻辑与异步 job runner 的衔接

`backend/internal/screenplay`
- 剧本领域模型
- YAML 组装、序列化、校验适配

`backend/internal/llm`
- Prompt 构造
- 模型调用抽象
- 响应解析和容错

`backend/internal/storage`
- 任务元数据、输入文本、生成产物持久化
- 样例和结果文件管理
- SQLite 和文件系统访问封装

`backend/internal/workflow`
- 章节到场景的业务规则
- 人物、地点、冲突、转场等结构化编排策略

`backend/internal/httpx`（建议后续新增）
- 请求解码与响应编码
- 错误映射
- 中间件装配

`frontend/`
- 输入作品与配置
- 展示任务状态
- 浏览、编辑和导出 YAML

`docs/`
- 范围、架构、进度、Schema、协作约束的权威来源

## 用户路径

核心用户：
- 想把小说快速改编成剧本初稿的作者或编辑

首版核心路径：
1. 输入作品信息和 3 章以上小说内容
2. 选择改编风格或补充要求
3. 提交生成任务
4. 查看生成进度和中间提示
5. 查看结构化剧本 YAML
6. 手动调整并导出

## 成功标准

产品成功：
- 能稳定产出合法 YAML
- YAML 足够结构化，便于继续编辑和程序消费
- 生成结果能体现章节到场景的清晰映射

工程成功：
- 目录结构清晰
- 后端模块边界明确
- PR/commit 节奏合理
- 评委与 AI agent 能快速理解仓库

演示成功：
- 能讲清楚输入、管线、输出、编辑、导出
- 能说明为何选择该 YAML Schema
- 能展示后端设计不是“单接口包办一切”
- 能展示 Go 服务中的中间件、任务状态、结构化日志和校验链路
